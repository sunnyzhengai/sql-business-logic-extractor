# Architecture

This document is the **first thing to read** when navigating this
repository cold. It explains what each top-level directory is for, how
they fit together, and where to look for any given concern.

For the pipeline phase-by-phase guide, see `tools/PHASES.md`.

---

## What this repo is

A toolkit for **mining business logic and surfacing variance from
existing SQL** in healthcare BI environments.

The end goal: catalyze data governance by giving stewards and BI
developers a structured view of how the same concepts are defined
differently across reports, so they can ratify a single source of
truth and move toward certified data models in Fabric.

The pipeline ingests a corpus of SQL views (typically exported from
SSMS) and produces evidence packs for steward conversations:

```
SQL views  →  parse  →  unified graph  →  community detection  →
   steward-ready artifacts  →  human ratification  →  certified definitions
```

This is the substrate for the destination architecture:
**Govern → Model → Self-serve → AI-assisted self-serve.**

## The pipeline (tools/)

The pipeline has **seven phases**. Phase folders are named `pN0_<purpose>`
so the file-tree ordering reveals the pipeline order at a glance.

```
p10_extract     SQL files       →  corpus.jsonl (per-view parsed structure)
p20_index       corpus.jsonl    →  unified networkx graph + lexical index
p30_analyze     graph           →  community detection, primary-community
                                    assignments, similarity findings
p40_synthesize  analysis        →  steward-ready markdown artifacts
p50_present     synthesized     →  interactive HTML, visualizations, deck slides
p60_hitl        human input     →  structured annotation records
p70_feedback    annotations     →  updated graph state for next analyze pass
```

A `shared/` folder holds cross-phase utilities (parsing helpers,
schemas, file I/O). A `diagnostics/` folder holds inspection and
triage tools (not part of the pipeline -- e.g., validation experiments,
sanity checkers).

For details on each phase, see `tools/PHASES.md`.

## Reading order for a new contributor

1. **This file** (`ARCHITECTURE.md`) -- top-level orientation.
2. **`tools/PHASES.md`** -- phase-by-phase guide.
3. **`wiki/index.md`** -- domain knowledge if working on healthcare-specific code.
4. **`tools/p10_extract/__init__.py`** -- entry point of the pipeline.
5. Follow imports forward through each phase.
6. For one-off diagnostics: **`tools/diagnostics/validate_graph_pivot.py`**
   is the most heavily-commented file and a good place to learn the codebase
   patterns.

## Conventions

### Folder naming

- **Pipeline phases** are prefixed `pN0_` (gaps of 10, leaving room for
  inserted phases without renumbering). Phase folders sort in pipeline
  order in any file tree.
- **Cross-cutting utilities** live in `tools/shared/`, not in any
  individual phase folder.
- **Diagnostics** (validation, sanity checks, triage) live in
  `tools/diagnostics/`. They are NOT part of the pipeline.
- **Tests** live in a `tests/` subfolder of each phase
  (`tools/p10_extract/tests/`, etc.), plus top-level integration tests in `tests/`.

### Imports

Imports use the phase-prefixed path:

```python
from tools.p10_extract.parser import parse_view
from tools.p20_index.graph_builder import build_corpus_graph
from tools.p30_analyze.communities import detect_communities
```

The phase number is visible in every import line, so when scanning a
script you can see which phases it touches.

### Code style

- **Function-level docstrings + inline comments are required** in this
  project. This overrides the typical "well-named identifiers do the
  explaining" rule -- the codebase is meant to be readable by a Python
  learner.
- Prefer plain idioms over clever one-liners. Comprehensions only when
  they read more clearly than the equivalent loop.
- Explain Python-specific quirks (mutable defaults, `__init__.py`,
  decorators, etc.) when they appear.

### Output paths

- The pipeline writes to a configurable `output_dir`. Defaults in
  CLI tools point under `/tmp/` or the cwd for local runs.
- In Fabric, the typical convention is
  `/lakehouse/default/Files/<pipeline_run_name>/...`.

## Where the auto-generated artifacts come from

- **`graphify-out/`** is regenerated automatically by a post-commit hook
  (the `graphify` skill). It contains a code-AST graph of the repo,
  community structure, and a generated wiki. Do not edit by hand.
- **`graphify-out/wiki/`** is auto-generated documentation of the
  codebase itself. Distinct from `wiki/` (the human-curated domain wiki).

---

## What this repo is NOT

- **Not a runtime service.** Tools run on-demand, typically in Fabric
  notebooks against a lakehouse corpus.
- **Not an ML model.** The analysis is graph-theoretic and rule-based,
  not learned. (LLM integration is a future possibility but
  intentionally out of scope for the pilot.)
- **Not a Collibra connector.** Outputs are meant to be ratified
  manually by stewards before any catalog publication.
- **Not a SQL execution engine.** The pipeline parses SQL with sqlglot;
  it does not run queries.

## Status (2026-05-17)

The repo is **mid-restructure**. The pipeline phase folders
(`tools/pN0_*`) exist as skeletons after Phase 0 of the restructure.

The pre-restructure architecture had tools spread across ~20 sibling
folders in `tools/` (`extract_corpus/`, `similarity/`, `cohort_extract/`,
etc.). Those folders still exist and continue to work; they will be
migrated into the new phase structure incrementally over the next
several restructure phases (see `tools/PHASES.md` migration table for
the current state of each legacy tool).

A **graph-pivot validation experiment**
(`tools/diagnostics/validate_graph_pivot.py`) ran in May 2026 against
the user's real 130-view corpus and returned PASS. The restructure
implements the architecture that experiment validated.
