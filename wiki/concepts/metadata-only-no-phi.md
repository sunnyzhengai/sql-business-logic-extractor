## Definition

The extractor operates exclusively on **schema metadata and query logic** — SQL
view / stored-procedure definitions and the Clarity data dictionary (table
names, column names, and their descriptions). It never connects to a database
and never reads patient rows. The inputs it processes are not PHI.

## Why it matters here

This is the backbone of the LLM-mode governance story. When descriptions are
generated by an LLM (Tool 3 / Tool 4), what leaves the process is *table/column
names, dictionary descriptions, resolved expressions, and filter predicates* —
authored meaning, not patient data (see [[sql-as-definitional-moments]]). So the
HIPAA framing that gates *data* movement does not apply to this pipeline; the
relevant control is the separate confidentiality/IP question of whether source
code and schema may leave the building — a policy decision, not a PHI one (see
[[governance-vs-compliance-layers]]).

This distinction is exactly the [[data-vs-metadata-ownership]] fork made
operational: the tool lives entirely on the metadata side.

Practical consequences:
- A personal LLM key (OpenAI / Azure free trial) is sufficient to prove value on
  real view definitions, because no PHI is in scope.
- The one residual edge case is **hardcoded literals inside SQL** (e.g.
  `WHERE PAT_MRN = '...'`). The controlling assumption is that PHI is not
  hardcoded in SQL; teams adopting LLM mode should confirm that holds.
- The engineered (no-LLM) path remains the default for customers whose policy
  forbids *any* source/schema egress, preserving "nothing leaves the building"
  as a structural guarantee (vendor SDKs are lazy-imported only in LLM mode).

## Open questions

- Do we add an optional pre-flight scrubber that flags/strips string literals
  in SQL before an LLM call, to neutralize the hardcoded-literal edge case?
- For enterprise Azure OpenAI under a BAA, is the egress-policy sign-off
  per-customer, or can we ship a standard attestation describing exactly what
  metadata is transmitted?
